## 选型方法论：先说清楚我们怎么选

在做模型选型之前，先建立一个评估框架。宠物饮食咨询这个场景，对模型的要求和通用对话不一样——用户期待的不是"聊得开心"，而是"方案可信、结构清晰、对宠物安全负责"。

我们从四个维度来评估候选模型：

**① 通用能力**：语言理解、逻辑推理、长文本生成、结构化输出。

**② 宠物/健康垂域能力**：是否经过宠物医疗数据训练、是否具备宠物营养学知识、是否能识别宠物禁忌（比如"洋葱对狗有毒"这种常识不能出错）。

**③ 成本与效率**：API调用价格、输出速度、并发能力。宠食记是一个面向C端的免费工具（至少MVP阶段如此），成本直接影响能否持续运营。

**④ 部署与生态**：国内访问稳定性、与现有技术栈的兼容性、工具链完善度。

打分方式：每项满分5分，综合加权评分。通用能力和垂域能力各占30%，成本35%（因为C端产品对价格高度敏感），生态5%。

## 1. 宠物垂类大模型

### 1.1 绮算法（帕奇宠 / QIALGpet 系列）

**基本信息**：重庆绮算法科技有限公司，2022年成立，智谱Z基金（星连资本）千万元级战略投资。已连融三轮（Z基金 → 启赋资本 → 启赋+聚恒创投），是宠物垂直AI赛道融资最多的公司之一。

**核心产品**：
- **帕奇宠（PAWKEY LAND）**：面向医院端的宠物医生Agent + 面向宠主端的宠物管家Agent
- NPU芯片宠物大模型加速卡：嵌入猫砂盆、饮水机等IoT设备，做边缘计算
- AI智能项圈19克、AI喂食器、AI ICU（已在30+医院落地）

**技术路线**：
- QIALGpet-T：自训练语义理解模型
- QIALGpet-B：自训练决策模型
- Relation RAG：专业医疗AI数据库（千万份宠物病历、医学影像、行为数据）
- 2025年已落地4城50+医院，辅助问诊服务超百万次，合作医院超200家

**对宠食记的参考价值**：绮算法是目前国内宠物垂域模型的技术标杆。它的技术路线——"通用底座 + 宠物医疗RAG + 垂域微调"——可以被我们借鉴。但它的定位偏医疗诊断，和我们的"饮食决策"定位有交叉但不直接竞争。值得关注的是，它已经被智谱系资本加持，意味着智谱的大模型在宠物垂域有生态布局。

**直接使用可行性**：低。绮算法的API目前主要面向B端宠物医院，没有公开的C端开发者API。如果未来它开放API，可以作为我们对话系统中"宠物医疗知识"的增强模块。

---

### 1.2 汪喵灵灵（菲鹏集团）

**基本信息**：菲鹏集团旗下宠物医疗AI产品，2024年7月上线，核心技术是自研「百目魔君」大模型，并在2025年3月接入DeepSeek进行强化。

**亮点数据**：
- PDEM宠医问诊横向测评综合评分第一
- 数十万用户基础
- 2025年3月31日开放API平台

**技术路线**：通用底座（DeepSeek）+ 宠物垂域SFT/RL微调，强化CoT（思维链）可解释性。

**对宠食记的参考价值**：汪喵灵灵验证了一个重要假设——**通用模型经过宠物垂域微调，可以在宠物问诊场景达到甚至超过专用模型的表现**。它的API开放对我们有潜在价值，但产品定位（在线问诊）和我们不同。

---

### 1.3 未见科技 / 宠有灵犀

**未见科技**：研发Pemi多模态大模型，专攻猫狗行为/情绪识别（数百种行为 + 数十种情绪），APP预计2026春节前后发布。技术路线偏视觉多模态和具身智能。

**宠有灵犀**：AI健康管理平台，基于多模态+区块链宠物身份，覆盖1200+疾病，诊断准确率92%，鼻纹识别99.3%。2025年4月启动天使轮融资，估值2500万。

**对宠食记的参考价值**：这两家更偏"通用健康管理平台"，多模态能力（便便识别、皮肤病变识别）是它们的核心差异点。对宠食记来说，多模态能力在菜谱识别（拍照识食材）和宠物状态识别（拍照判断体态）上是加分项，但MVP阶段不做。

### 宠物垂类模型小结

| 维度 | 绮算法/帕奇宠 | 汪喵灵灵 | 未见科技 | 宠有灵犀 |
|------|:--:|:--:|:--:|:--:|
| 宠物垂域深度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| API可用性 | 不可用(B端) | ✅ 2025.3开放 | 未知 | 未知 |
| 饮食场景覆盖 | 弱(偏医疗) | 弱(偏问诊) | 弱(偏行为) | 弱(偏管理) |
| 对宠食记价值 | RAG数据源 | 技术路线验证 | 长期多模态储备 | 区块链ID参考 |

**结论**：目前的宠物垂类大模型都聚焦在"医疗诊断"而非"饮食决策"，没有一个可以直接满足宠食记的需求。但它们的存在证明了"通用模型+垂域数据"这一技术路线的可行性。

## 2. 通用大模型横向对比

### 3.2 最终推荐：分层策略

不是选一个模型，而是根据场景分层使用。

```
┌──────────────────────────────────────────┐
│  第一层：主力模型                          │
│  DeepSeek V3（或 V3.1）                    │
│  → 承担 90% 的日常饮食咨询对话             │
│  → 已实测验证，成本可控，质量达标            │
├──────────────────────────────────────────┤
│  第二层：降级备选                          │
│  Qwen-Plus 或 GLM-4-Flash（免费）           │
│  → 高并发时降级分流                        │
│  → 简单问题（食材能不能吃、喂食量建议等）     │
│  → GLM-Flash成免费，无成本压力              │
├──────────────────────────────────────────┤
│  第三层：长文本增强                        │
│  Kimi K2（128K上下文）                     │
│  → 体检报告分析、长病历解读                 │
│  → 按需调用，不日常使用                     │
├──────────────────────────────────────────┤
│  第四层：质量评估（长期建设）                │
│  Claude Haiku 4.5 / GPT-4o-mini           │
│  → 定期抽查输出质量                        │
│  → 作为Prompt迭代的评测基线                │
├──────────────────────────────────────────┤
│  第五层：垂域知识增强（长期建设）            │
│  绮算法/汪喵灵灵 API + 自建宠物饮食RAG      │
│  → 当产品积累足够用户数据后                 │
│  → 自建宠物饮食知识库 + RAG增强            │
│  → 降低对通用模型宠物知识的依赖              │
└──────────────────────────────────────────┘
```

### 3.3 为什么DeepSeek是MVP阶段的最优解

1. **我们已经跑通了**：Demo用的就是DeepSeek，不需要更换技术栈
2. **成本可控**：一次典型的食谱咨询对话（用户提问 + AI回复），大约消耗2000-3000 tokens。按输出¥8/百万tokens算，一次对话的API成本约¥0.02-0.03。假设日活1000人、每人每天5轮对话，日成本不超过¥150，月成本约¥4500
3. **国内稳定**：不需要翻墙，延迟低，合规风险小
4. **结构化输出能力达标**：实测表明DeepSeek能稳定生成带表格、列表、加粗的结构化食谱
5. **涨价风险可控**：DeepSeek正在从"价格屠夫"转向"合理定价"（2025年9月涨价到¥12/百万tokens输出），但即使涨了，成本也在可接受范围内

### 3.4 降级策略

宠食记作为C端产品，需要保证服务可用性。当DeepSeek API不可用时（高峰限流、维护等），自动切换到备用模型：

- **优雅降级路径**：DeepSeek → GLM-4-Flash（免费，能力弱但可用）→ Qwen-Plus
- **降级时机**：API响应时间超过5秒、错误率超过5%、并发限制触发
- **用户无感**：降级时用户不需要知道模型换了，只是回答质量可能有轻微下降
