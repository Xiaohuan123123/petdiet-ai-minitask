## 4. 商业判断

### 4.1 先说清楚一个根本问题：谁会为这个产品付钱？

做商业判断不能靠感觉。我们先把用户拆开看。

根据白皮书和已有分析，宠食记的潜在用户可以分为四类。每一类的付费逻辑完全不同：

| 用户层 | 典型画像 | 对宠食记的核心需求 | 付费意愿 | 付费逻辑 |
|------|------|------|:---:|------|
| **焦虑型新手** | 00后/90后，第一次养宠，遇到问题不知道怎么办 | "告诉我现在该怎么做" | ⭐⭐⭐ | 解决焦虑，为确定性付费 |
| **慢病/病后宠主** | 宠物确诊肾病/糖尿病/术后恢复中 | "帮我每天管好它的饮食" | ⭐⭐⭐⭐⭐ | 解决刚需，为省心付费 |
| **品质升级型** | 高收入城市白领，想做鲜粮/自制但不知道怎么配 | "像私人营养师一样服务" | ⭐⭐⭐⭐ | 品质追求，为专业付费 |
| **多宠管理型** | 2只以上宠物，品种/年龄/健康状况各不相同 | "帮我把这些全部管起来" | ⭐⭐⭐ | 效率诉求，为便利付费 |

**早期核心付费用户是第二类——慢病/病后宠主。**

为什么？第一个原因，白皮书数据已经告诉我们：6 岁以上猫不健康占比超 50%，9 岁以上犬不健康占比达 53.4%。这是一个庞大且持续扩大的群体。第二个原因，他们的付费决策链路最短——宠物刚确诊或刚做完手术，主人处于焦虑峰值，愿意为任何能"帮宠物好起来"的方案付费。第三个原因：这个场景目前市场上没有产品覆盖——兽医说"吃处方粮"，但不告诉你具体怎么吃、吃多久、怎么根据指标调整、什么时候可以恢复正常饮食。我们填补的，就是这个空白。

### 4.2 用户为什么付费？——不只是"功能好"，而是"解决了我的什么焦虑"

用户付费不是因为 AI 很酷，而是因为 AI 帮他们摆脱了某种困境。我们用 Jobs-to-be-Done 框架来描述：

**用户雇佣宠食记，是为了完成以下任务：**

| JTBD | 用户现状 | 宠食记提供的替代方案 | 为什么愿意付费 |
|------|------|------|------|
| **"帮我的宠物在生病后恢复健康"** | 兽医说完就走了，自己在小红书搜，越搜越焦虑 | AI 基于诊断结果生成详细饮食方案，每天跟踪，异常提醒 | 这是刚需。不付费的替代方案是"在社交媒体上盲目搜索"，太不可靠 |
| **"告诉我，我的宠物每天该吃什么"** | 不知道该喂多少、不知道换粮对不对、怕喂错伤害宠物 | 输入宠物档案，AI 给出日常饮食建议 + 每周调整 | 这是高频需求。替代方案是"凭感觉喂"，但对追求科学养宠的人不满足 |
| **"帮我的宠物减肥/控制慢病饮食"** | 兽医说"少吃点"，但没有可持续的执行方案 | 设定目标体重，AI 每周调整喂食量 + 跟踪进展 | 这是持续需求。替代方案是"自己算热量"，大部分人不愿意做 |
| **"给我一份安全、营养均衡的鲜粮食谱"** | 想做鲜粮但不知道配比，怕缺营养 | AI 生成完整食谱：食材清单、分量、步骤、营养补充提示 | 这是品质需求。替代方案是"网上找的食谱"，没有个性化 |

**一句话总结：用户是在为"放心"付费，不是在为"功能"付费。** 他们愿意花钱的原因不是 AI 能生成食谱——这个 Google 也能做——而是 AI 生成的食谱是**针对它的宠物定制的**，而且**有人告诉我为什么这样配**。

### 4.3 定价策略

#### 4.3.1 定价原则

1. **低门槛切入**：让用户第一次付费没有心理负担。9.9 元和 199 元是两个完全不同的决策。先让用户花 9.9 元解决一个具体问题，觉得值了，再推更贵的
2. **价值可感知**：付费的那一刻，用户立刻看到区别于免费版的体验差异
3. **锚定参照系**：让用户觉得"对比去医院/请营养师，这个太便宜了"
4. **MVP 阶段宁可定价偏低**：先验证"有人愿意付"，再优化"怎么让更多人付更多"

#### 4.3.3 定价逻辑

单次付费的参照系：宠物医院一次挂号费 20-80 元，一次营养师咨询 100-300 元。所以 ¥9.9-19.9 的单次付费在用户心里是"比去一次医院便宜多了"的定位。这个定价策略的核心是**让用户觉得"先试一下，反正也不贵"**。

月度订阅的参照系：一份宠物保险月均 30-80 元，一袋中端狗粮 200-400 元/月。¥29/月的订阅在用户心里是"一杯奶茶的价格换一个私人营养师"。

#### 4.3.4 定价在 MVP 阶段的风险与验证点

| 假设 | 风险 | 如何验证 |
|------|------|---------|
| "用户愿意为病后方案付 ¥19.9" | 价格可能偏低，但先验证"有需求"更重要 | 1周内看转化率 |
| "用户愿意为订阅付 ¥29/月" | 月度订阅的付费障碍比单次高很多 | 先推单次，订阅从"已付费用户"中转化 |
| "免费版每天3轮足够" | 高频用户可能觉得不够，导致流失 | 观察实际使用频次，两周后决定是否调整 |

### 4.4 获客策略

#### 4.4.1 第一阶段：冷启动（前 2 周 · 验证阶段）

**核心思路**：不花钱投广，用内容撬动自然流量。

**具体做法**：

**渠道一：小红书 —— 用"真实案例"种草**

在小红书建一个"宠食记"账号，不做硬广，而是用真实案例驱动的知识内容：
- 格式：宠物档案卡片 + AI 生成的食谱截图 + 一段点评
- 选题举例："3岁金毛软便期饮食方案（附完整食谱）""12kg柯基一顿该吃多少？AI算出来了"
- 每个帖子末尾引导用户"想给你的宠物也生成专属食谱？评论区告诉我它的品种体重，我帮你问AI"
- 预期：每天 2-3 条内容，两周内目标积累 30-50 个真实评论追问

---

**渠道二：宠物社群（微信群/QQ群）—— 用"免费工具"换用户**

找到 5-10 个活跃的宠物交流群（金毛群、柯基群、铲屎官群等），不直接打广告，而是：
- 看到群里有人问"我家XX拉肚子怎么办""换什么粮好"这类问题时，用宠食记帮TA生成一个回答，截图发群里
- 话术："我用一个AI工具帮你查了一下，你看看这个方案有没有参考价值。如果需要我可以帮你也生成一份"
- 这是在提供真实价值，不是推广告。用户自然会问"这是哪个APP？"

---

**渠道三：宠物医院/宠物店合作（低成本 BD）**

找到 2-3 家本地宠物医院或宠物店，和老板聊一个简单的合作：
- 在医院/店里放一个二维码立牌，文案："出院后不知道怎么喂？AI 帮你做饮食方案。扫码免费试用"
- 给医院提供一个"患者饮食管理工具"的概念，帮兽医做他们没时间做的随访
- 不需要付钱，这是帮医院/店里提供增值服务

---

**目标（两周内）**：

| 指标 | 目标值 | 说明 |
|------|:---:|------|
| 总注册用户 | 200-500 人 | 来自上述三个渠道 |
| 小红书账号粉丝 | 200-500 | 用内容自然积累 |
| 社群覆盖人数 | 1000+ | 5-10个群 |
| 合作医院/店 | 2-3 家 | 协议达成即可 |
| **DAU** | **30-50** | 这是核心指标，持续使用的才算"用户" |
| **对话完成率** | **>60%** | 用户发起对话后，收到完整AI回复的比例 |
| 付费尝试率 | >5% | 至少尝试一次付费 |

最关键的不是注册量，而是**有多少人用了之后第二天还会来**。冷启动阶段，次日留存才是真正的验证。

## 5. 两周验证计划

### 5.1 验证什么？

两周时间很短，所以必须聚焦。别想着两周内把产品做完美，也别想着两周内就有一千用户。

**我们最需要验证的三个核心假设：**

| 假设 | 验证指标 | 为什么这个最重要 |
|------|---------|------|
| **H1：用户会主动问宠物饮食问题** | 日均对话数 > 10 条/活跃用户 | 如果用户不知道问什么，产品就废了 |
| **H2：用户对AI饮食方案的信任度达标** | 方案采纳率 > 50%（用户反馈"方案有用"） | 信任是付费的前提 |
| **H3：有人愿意为饮食方案付钱** | ≥2 笔付费订单 | 不是"很多人愿意付"，而是"至少有一个人真的掏了钱" |

### 5.2 每周做什么

#### Week 1（第 1-7 天）：把产品推出去，看有没有人用

| 天 | 做什么 | 具体产出 | 资源需求 |
|:---:|------|------|------|
| Day 1 | 部署线上版 + 创建小红书账号 + 加入 5 个宠物社群 | 产品可访问、账号就绪 | 服务器（已有）|
| Day 2 | 发布 2 篇小红书内容 + 在社群中首次提供 AI 回答（不推产品，先提供价值）| 2 篇帖子 + 3 条社群回复 | 运营时间 2h |
| Day 3 | 发布 2 篇内容 + 社群互动 + 联系 3 家本地宠物医院/宠物店谈合作 | 2 篇帖子 + 3 个合作洽谈 | 运营时间 3h |
| Day 4 | 发布 2 篇内容 + 社群互动 + 跟进医院合作 | 2 篇帖子 + 合作推进 | 运营时间 2h |
| Day 5 | 发布 2 篇内容 + 分析前五日数据：多少人访问了？多少人对话了？ | 2 篇帖子 + 数据简报 | 运营 1h + 分析 1h |
| Day 6 | 发布 2 篇内容 + 根据数据反馈调整 Prompt 或 UI | 内容 + 产品微调 | 运营 1h + 开发 2h |
| Day 7 | **Week 1 复盘**：写出 Week 1 验证报告 | 一份文档 | 分析 2h |

**Week 1 判断标准**：如果有人主动用产品对话了 > 5 轮，说明需求真实存在。如果没有一个人对话超过 3 轮，说明获客方式或产品形态有问题，需要调整。

#### Week 2（第 8-14 天）：验证付费 + 优化留存

| 天 | 做什么 | 具体产出 | 资源需求 |
|:---:|------|------|------|
| Day 8 | 从 Week 1 用户中访谈 3-5 人：你为什么用？你觉得哪里不够好？ | 3-5 份访谈记录 | 运营时间 3h |
| Day 9 | 根据反馈快速迭代产品（修 bug + 改 Prompt + 优化输出格式）| 产品 v1.1 | 开发时间 4h |
| Day 10 | 针对最活跃的用户，主动推送一次付费方案（"要不要试试定制食谱？"）| 首次付费转化尝试 | 运营 1h |
| Day 11 | 新媒体内容持续发布 + 社群互动 | 2 篇内容 + 社群互动 | 运营 2h |
| Day 12 | 分析付费转化数据 + 继续迭代 Prompt | 数据简报 + 产品 v1.2 | 分析 1h + 开发 2h |
| Day 13 | 用户访谈 round 2 + 继续迭代 | 访谈记录 + 产品微调 | 运营 2h + 开发 1h |
| Day 14 | **两周总结报告** | 完整验证报告 | 分析 4h |

**Week 2 判断标准**：
- 如果产生了 ≥2 笔付费：验证通过，继续推进
- 如果 0 付费但有用户表示愿意：价格或付费场景需要调整
- 如果用户活跃但不愿意付费：免费版的体验可能"太好了"，需要重新设计免费/付费边界

### 5.3 需要什么资源

| 资源类型 | 具体内容 | 优先级 |
|------|------|:---:|
| **人力：产品/运营 1 人** | 写内容、管社群、做 BD、做用户访谈、分析数据。全职或专注投入 20h/周 | P0 |
| **人力：开发 0.5 人** | 改 bug、调 Prompt、改 UI。不需要全职，但要在产品出问题时能改 | P0 |
| **服务器** | DeepSeek-V4-Pro API（已有，成本可忽略）+ 部署服务（已有 localhost，线上化需一个便宜的云服务器） | P1 |
| **域名** | 一个域名，用于避免用户看到 IP 地址。¥50/年 | P2 |
| **营销预算** | ¥0。冷启动阶段不投钱 | — |
| **总计预算** | **< ¥500**（仅域名+云服务器基础版，如果已有可复用则 < ¥100） | |

### 5.4 两周后的决策框架

两周结束后，根据数据做三种决策：

```
┌────────────────────────────────────────────┐
│  绿灯：继续推进                              │
│                                            │
│  信号：DAU > 30，日对话 > 5 轮/活跃用户，    │
│        有 ≥2 笔付费，用户访谈正面反馈明确     │
│                                            │
│  行动：                                    │
│  1. 加大内容获客力度（从小红书 2 篇/天 → 5 篇）│
│  2. 扩展医院合作（从 2-3 家 → 10 家）        │
│  3. 启动订阅功能开发                        │
│  4. 准备种子/天使轮融资材料                 │
├────────────────────────────────────────────┤
│  黄灯：调整方向                              │
│                                            │
│  信号：有活跃用户但付费转化低，或             │
│        获客快但留存差                         │
│                                            │
│  行动：                                    │
│  1. 深度访谈用户，找到"不付费"的本质原因     │
│  2. 重新设计免费/付费边界                   │
│  3. 给一周时间验证调整后的方案               │
├────────────────────────────────────────────┤
│  红灯：重新思考                              │
│                                            │
│  信号：无人主动使用，或                     │
│        用户明确表示"不需要这个产品"           │
│                                            │
│  行动：                                    │
│  1. 反思是获客方式有问题，还是产品本身需求不够 │
│  2. 如果是前者，换获客渠道再试一周           │
│  3. 如果是后者，考虑 Pivot                   │
└────────────────────────────────────────────┘
```

### 5.5 心态上的一件事

两周验证的目的不是"证明我是对的"，而是"找到什么是对的"。

如果你花了大量精力做一个功能但用户不用，这不算失败——这是以最低成本获得了一个重要的市场信号。两周的时间成本远低于"花三个月做一个没人用的产品"。

所以这两周的核心心态是：**快做、快看结果、快调整。别烧钱，烧认知。**
